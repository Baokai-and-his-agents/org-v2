# Security Policy

## 🔒 报告安全漏洞

我们非常重视 org-v2 的安全性。如果你发现安全漏洞，请负责任地报告。

### 请勿公开披露

**请不要**在 GitHub Issues、Discussions 或其他公开渠道报告安全漏洞。

### 如何报告

请通过以下方式私下报告安全问题：

1. **GitHub Security Advisories**（推荐）
   - 访问 https://github.com/Baokai-and-his-agents/org-v2/security/advisories
   - 点击 "Report a vulnerability"
   - 填写详细信息

2. **邮件**
   - 发送至：[维护者邮箱]（待添加）
   - 主题：`[SECURITY] [org-v2] 简短描述`

### 报告应包含

为了帮助我们更快地理解和解决问题，请提供：

- 漏洞的详细描述
- 重现步骤（越详细越好）
- 受影响的版本
- 可能的影响范围
- 你建议的修复方案（如果有）
- 你的联系方式

### 响应时间

- **确认**：收到报告后 48 小时内确认
- **初步评估**：7 天内给出初步评估
- **修复计划**：根据严重程度，30-90 天内发布修复

### 披露政策

- 我们会在修复发布后公开披露漏洞
- 我们会在安全公告中感谢报告者（除非你要求匿名）
- 我们遵循负责任的披露原则

## 🔐 安全最佳实践

使用 org-v2 时，请遵循以下安全最佳实践：

### 1. 保护敏感信息

**不要**在能力文件或任务中存储：
- API 密钥
- 密码
- 访问令牌
- 个人身份信息

**使用**环境变量或专用的密钥管理工具。

### 2. 定期更新

- 定期检查更新：`git pull`
- 查看 [CHANGELOG.md](CHANGELOG.md) 了解安全修复
- 订阅 GitHub Security Advisories

### 3. 审查脚本

在运行任何脚本前：
- 阅读脚本内容
- 理解它做什么
- 确认来源可信

### 4. 权限控制

- 使用最小权限原则
- 不要以 root 运行脚本（除非必要）
- 审查文件权限

### 5. 备份

- 定期备份你的组织数据
- 使用版本控制（Git）
- 定期 push 到远程仓库

## 🚨 已知安全考虑

### Shell 脚本执行

org-v2 使用 Bash 脚本自动化任务。脚本执行本质上有风险。

**我们的措施**：
- 所有脚本使用 `set -e`（遇错退出）
- 输入验证
- 避免 `eval` 和动态执行

**你的责任**：
- 审查任何脚本后再运行
- 不要执行来自不受信任来源的脚本

### 文件权限

某些脚本需要执行权限（`chmod +x`）。

**注意**：
- 只对必要的脚本添加执行权限
- 检查脚本内容
- 使用 Git 追踪权限变更

### 第三方依赖

org-v2 本身没有依赖项（纯 Bash + Markdown）。

如果你添加了依赖：
- 使用固定版本
- 审查依赖的安全性
- 定期更新依赖
