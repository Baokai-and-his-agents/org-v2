# Skill: 问题驱动开发（PDD - Problem-Driven Development）

**类型**：skill  
**创建**：2026-06-12  
**更新**：2026-06-12  
**复用次数**：1

---

## 🎯 用途

当遇到技术问题时，系统化地调试、修复并沉淀为可复用知识，避免同样的问题再次发生。

---

## 📥 输入

- 问题描述（错误信息、失败日志）
- 期望行为
- 实际行为

---

## 📤 输出

- 根本原因分析
- 修复方案（代码/配置）
- Knowledge 文档（防止再次发生）
- 测试验证记录

---

## 🔄 使用流程

### 1. 问题重现
- [ ] 记录完整错误信息
- [ ] 确认触发条件
- [ ] 本地复现问题

### 2. 根因分析
- [ ] 阅读错误日志
- [ ] 检查相关代码/配置
- [ ] 识别差异（期望 vs 实际）
- [ ] 找到根本原因（不是表象）

### 3. 修复实施
- [ ] 设计最小修复方案
- [ ] 实施修复
- [ ] 本地测试验证
- [ ] 提交修复

### 4. 知识沉淀
- [ ] 写 knowledge 文档
- [ ] 记录根因、修复方案、预防措施
- [ ] 更新相关 skill/data
- [ ] 注册到 capabilities/INDEX.md

---

## ✅ 检查清单

### 分析阶段
- [ ] 问题描述清晰完整
- [ ] 已尝试本地复现
- [ ] 已找到根本原因（非表象）

### 修复阶段
- [ ] 修复方案最小化（不过度设计）
- [ ] 本地测试通过
- [ ] 不引入新问题

### 沉淀阶段
- [ ] 创建 knowledge 文档
- [ ] 包含：问题、原因、方案、预防
- [ ] 更新 INDEX.md

---

## 📝 示例

### 成功案例

**问题**：GitHub Actions duty-check 失败

**根因分析**：
1. 错误信息：`sed: can't read ... No such file or directory`
2. 检查代码：发现 macOS 特有的 `sed -i ''` 语法
3. 环境差异：GitHub Actions 运行在 Linux
4. 根本原因：跨平台兼容性问题

**修复方案**：
- 方案 A：添加 OS 检测 + 条件分支（复杂）
- 方案 B：直接在 heredoc 中使用变量替换（简单）
- **选择 B**：更简单、更可靠

**知识沉淀**：
- 创建 `knowledge/cross-platform-bash-compatibility.md`
- 记录 macOS vs Linux 差异
- 提供通用解决方案
- 防止类似问题再发生

**时间**：从发现到修复 15 分钟

## 📌 维护说明

- 更新频率：每次遇到新问题类型时
- 负责人：任何遇到问题的 Agent
- 典型模式：问题 → 分析 → 修复 → 沉淀

---

## 🎓 关键原则

1. **问题即机会**
   - 每个问题都是沉淀知识的机会
   - 不要只修复，要沉淀

2. **找根因，不治标**
   - 不要只看表象
   - 深入到根本原因

3. **最小修复**
   - 不过度设计
   - 优先简单可靠的方案

4. **必须沉淀**
   - 修复后必须创建 knowledge
   - 防止再次发生
