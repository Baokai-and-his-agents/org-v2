# Skill: Launch Preparation

**类型**：skill  
**创建**：2026-06-12  
**更新**：2026-06-12  
**复用次数**：1（task 007）

---

## 🎯 用途

为产品、项目或功能准备外部发布材料，包括清晰的价值主张、发布文案、FAQ、反馈机制，确保顺利获得第一批外部用户。

---

## 📥 输入

- 产品/项目信息（名称、功能、定位）
- 核心数据（性能、效果、验证结果）
- 目标用户画像
- 发布渠道（Twitter/HN/Blog 等）

---

## 📤 输出

- 发布文案（多渠道版本）
- FAQ 文档
- Issue/反馈模板
- 反馈收集机制
- 发布计划时间表

---

## 🔄 使用流程

### 1. 价值主张提炼

**核心问题**：
- 你解决什么问题？
- 为什么现有方案不够好？
- 你的方案有何不同？
- 有什么数据支持？

**提炼公式**：
```
价值主张 = 核心问题 + 独特解决方案 + 可验证结果
```

**示例**：
- 问题：AI 组织能力沉淀率低（14%）
- 方案：强制沉淀机制
- 结果：8.4 倍效率提升（117% vs 14%）

### 2. 撰写发布文案

#### Twitter 文案（280 字符以内）

**结构**：
1. Hook（1 句话吸引注意）
2. 问题（1 句话）
3. 方案（1 句话）
4. 数据（1 个核心数字）
5. CTA（行动号召）

**示例模板**：
```
🚀 [Hook：惊人发现/成果]

大多数 [target] 遇到 [problem]。

[Your solution] 通过 [mechanism] 解决了这个问题。

结果：[X倍/X%] 的 [improvement]。

🔗 [link] #hashtag
```

#### HackerNews 文案（Show HN）

**标题**：`Show HN: <项目名> - <一句话价值主张>`
- ≤ 80 字符
- 清晰、具体、吸引人

**正文结构**：
1. **问题陈述**（2-3 段）
   - 当前痛点
   - 为什么重要
   - 现有方案的不足

2. **解决方案**（2-3 段）
   - 你的方法
   - 核心创新
   - 技术选择

3. **验证结果**（1-2 段）
   - 真实数据
   - 测试场景
   - 成功案例

4. **开放性问题**（1 段）
   - 你不确定的
   - 寻求反馈的
   - 引发讨论的

5. **行动号召**
   - GitHub 链接
   - Quick Start 指引
   - 反馈渠道

**风格**：
- 诚实（承认局限）
- 技术性（数据说话）
- 开放（欢迎讨论）

### 3. 创建 FAQ

**问题来源**：
- 用户会问什么？
- 你自己有什么疑问？
- 竞品用户问过什么？

**分类结构**：
1. **基础概念**（是什么）
2. **对比选择**（vs 竞品）
3. **适用性**（适合谁）
4. **使用方式**（怎么用）
5. **技术细节**（如何实现）
6. **效果验证**（真的有用吗）
7. **贡献支持**（如何参与）

**每个问题**：
- 简洁的问题标题
- 清晰的答案（2-5 句话）
- 示例或数据（如适用）
- 相关链接

**数量**：
- 最小：15 个问题
- 理想：30 个问题
- 覆盖 90%+ 疑问

### 4. 设置反馈机制

#### Issue 模板

**至少 3 种**：
1. Bug Report
2. Feature Request
3. Question/Help

**每个模板**：
- 清晰的字段
- 引导性问题
- 检查清单
- 标签建议

#### Discussions

**分类建议**：
- 💬 General（一般讨论）
- 💡 Ideas（建议）
- 🙏 Q&A（问答）
- 🎉 Show and Tell（展示）

#### 其他渠道

- Twitter hashtag（可选）
- Discord/Slack（可选）
- Email（最后手段）

### 5. 准备 Demo

**快速开始视频**（可选）：
- ≤ 5 分钟
- 演示核心功能
- 从零到结果

**文字版（必需）**：
- QUICKSTART.md
- 10 分钟内完成
- 每步清晰

## 💡 最佳实践

### 1. 数据胜于概念

❌ "我们非常快"  
✅ "8.4 倍效率提升"

具体数字让人信服。

### 2. 诚实胜于夸大

承认局限：
- 样本量小
- 早期阶段
- 需要反馈

这比过度承诺更可信。

### 3. 故事胜于功能

不要罗列功能，讲一个故事：
- 遇到问题
- 尝试解决
- 得到结果
- 学到教训

### 4. 行动胜于等待

不要等"完美"才发布：
- 准备 80% → 发布
- 收集反馈 → 改进
- 再次发布

完美是迭代出来的。

## 🎓 关键原则

1. **清晰 > 完整** - 10 个问题回答好 > 50 个问题回答模糊
2. **数据 > 主张** - 让数字说话
3. **开放 > 完美** - 承认不足，邀请反馈
4. **行动 > 等待** - 准备好 80% 就发布
