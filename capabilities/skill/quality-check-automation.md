## 🎯 用途

为目录或资产集合创建自动化质量检查系统，识别不一致、缺陷或过期内容，生成可操作的质量报告。

---

## 📥 输入

- 目标目录路径
- 质量标准定义
- 检查规则列表
- 报告输出路径

---

## 📤 输出

- 质量检查报告（Markdown）
- 问题列表（按优先级）
- 改进建议
- 健康度评分

---

## 🔄 使用流程

### 1. 定义质量标准

明确什么是"合格"：
- [ ] 格式要求（必需字段、结构）
- [ ] 内容要求（完整性、准确性）
- [ ] 时效性要求（更新频率）
- [ ] 一致性要求（命名、风格）

### 2. 设计检查规则

**常见检查类型**：

**数量同步**：
- 实际文件数 vs 声明数量
- 索引条目 vs 实际文件

**格式完整性**：
- 必需字段存在性
- 文档结构一致性
- 引用链接有效性

**过期检测**：
- 最后修改时间
- 未使用资产标记

**一致性**：
- 命名规范
- 版本号格式

### 3. 实现检查脚本

**推荐工具**：
- Bash（简单场景）
- Python（复杂逻辑）
- Node.js（JSON 处理）

**脚本结构**：
```bash
#!/bin/bash
set -e

# 1. 收集数据
collect_data() {
    # 统计实际情况
}

# 2. 执行检查
run_checks() {
    # 对比标准
}

# 3. 生成报告
generate_report() {
    # 输出结构化报告
}

# 主流程
main() {
    collect_data
    run_checks
    generate_report
}

main
```

### 4. 定义报告格式

**推荐结构**：
```markdown
# 质量检查报告

**时间**：<timestamp>

## 📊 统计

| 类型 | 实际 | 期望 | 状态 |
|------|------|------|------|
| ... | ... | ... | ✅/⚠️ |

## 🔍 问题

- ⚠️ <path>: <issue>
- ❌ <path>: <critical-issue>

## ⏰ 过期检测

<outdated-list>

## 📋 总结

✅/⚠️ **健康状态**：<excellent/good/needs-improvement>

<summary-and-suggestions>
```

### 5. 集成到工作流

- PR 提交前（本地）

**自动修复**（可选）：

## 💡 最佳实践

### 1. 渐进式构建

不要一次实现所有检查：
- 从最重要的开始
- 每次添加一个检查
- 逐步完善

### 2. 人类可读的报告

报告给人看，不是给机器：
- 使用 emoji（✅ ⚠️ ❌）
- 清晰的表格
- 可操作的建议

### 3. 避免过度检查

太严格 = 没人用：
- 优先检查高价值规则
- 允许合理偏差
- 区分错误和警告

### 4. 自动修复优先

能自动修复就别让人做：
- 格式化
- 索引同步
- 简单修正

## 🎓 关键原则

1. **质量标准先行** - 先定义"好"，再检查
2. **自动化节省时间** - 手动 30 分钟 → 自动 5 秒
3. **报告推动改进** - 不只发现问题，还要给建议
4. **持续运行** - 检查不是一次性，是持续保障
