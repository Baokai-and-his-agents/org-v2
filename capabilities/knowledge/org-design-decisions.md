## 🎯 背景

从 org-blueprint 分析中发现的关键问题，指导 org-v2 的设计决策。

---

## 📊 org-blueprint 核心问题

### 问题 1：概念密度过高
- 5 层架构、4 级自主等级、228 行文档
- 14 个任务只产生 2 个 skill（14% 转化率）
- 新 agent 需要读数千行文档

### 问题 2：文档 vs 实际运行脱节
- Duty Agent 设计完整但可能未运行
- Skillify 循环未形成闭环
- 缺少度量可见性

### 问题 3：产品边界模糊
- 文档说"不是产品仓库"，但有完整产品代码
- Sub2API 有 7 个文件但标记为"暂停"

---

## 💡 org-v2 设计决策

### 决策 1：3 层结构而非 5 层

**选择**：Control / Ops / Capabilities（3 层）

**拒绝**：Owner / Agent / 能力 / 系统 / Duty（5 层）

**原因**：
1. 1 人 + agents 不需要企业级复杂度
2. 减少 60% 认知负担
3. 新 agent 10 分钟上手 vs 1 小时

**权衡**：
- 丢失：详细的分层治理
- 获得：快速上手、易于维护

---

### 决策 2：NOW.md 单一入口

**选择**：一个文件显示所有当前状态

**拒绝**：15 步入口、多文件查找

**原因**：
1. 10 秒内知道"现在做什么"
2. 单一事实来源
3. 可自动更新

**权衡**：
- 丢失：详细分类导航
- 获得：极低认知负担

---

### 决策 3：强制能力沉淀评估

**选择**：每个任务结束必须回答"能否复用"

**拒绝**：自愿沉淀、无强制检查

**原因**：
1. 提高沉淀率从 14% 到 30-50%
2. Agent 可替换，能力才是护城河
3. 闭环驱动持续改进

**机制**：
- 任务文档模板包含"待沉淀能力"区域
- 工作日志必须记录沉淀评估
- 周报显示沉淀率

---

### 决策 4：度量优先

**选择**：自动周报、能力资产统计、沉淀率追踪

**拒绝**：手动复盘、定性评估

**原因**：
1. 无法度量 = 无法改进
2. 自动化减少 Owner 负担
3. 数据驱动迭代

**工具**：
- weekly-report.sh 自动生成
- GitHub Actions 定时运行
- INDEX.md 实时统计

---

### 决策 5：产品边界严格

**选择**：总部只保留 portfolio + 1 页 brief

**拒绝**：产品代码、部署配置、详细文档

**原因**：
1. 总部 = 索引 + 战略
2. 产品 = 独立仓库（权限、CI/CD 隔离）
3. 避免边界模糊

**执行**：
- products/ 只有 portfolio.md
- 产品代码在 github.com/org/product-name

---

## 📐 设计原则

从这些决策中提炼的核心原则：

1. **极简优于完美**
   - 能跑的简单系统 > 设计精美的复杂系统
   - 50 行 README > 139 行 README

2. **可执行优于可设计**
   - 先跑真实任务，再沉淀能力
   - 不提前设计未验证的技能

3. **度量驱动改进**
   - 每周自动生成数据
   - 沉淀率、完成率、复用率

4. **强制闭环**
   - 任务 → 评估 → 沉淀 → 复用
   - 无闭环 = 无改进

5. **边界清晰**
   - 总部 vs 产品
   - Owner vs Agent
   - 文档 vs 代码

---

## 🔄 验证方式

**3 个月后检查**：
1. 沉淀率 ≥ 30%？
2. 新 agent 10 分钟上手？
3. Owner 每周时间下降？
4. 能力资产数量上升？

**如果否 → 重新评估决策**
