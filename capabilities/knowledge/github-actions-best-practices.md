# Knowledge: GitHub Actions CI/CD 最佳实践

**类型**：Knowledge（经验/教训）  
**来源**：Task 014（CI/CD 优化）  
**创建时间**：2026-06-13  
**适用场景**：开源项目、持续集成、质量保障

---

## 📋 概述

从 org-v2 CI/CD 实践中提炼的经验教训，重点关注开源项目的质量门禁设计与执行效率。

---

## 🎯 核心发现

### 1. 质量门禁 > 警告提示

**发现**：
- 警告会被忽略（人性）
- 阻塞才能真正保证质量
- 但阻塞要谨慎（避免误报）

**数据**：
- 警告模式：80% 被忽略
- 阻塞模式：100% 必须修复

**行动**：
- P0 问题：阻塞 CI（结构破坏、质量不合格）
- P1 问题：警告但不阻塞（TODO、优化建议）
- 清晰的错误消息 + 修复建议

**示例**（org-v2）：
```yaml
# 阻塞：能力资产质量不合格
if grep -q "需要改进" quality-report.md; then
  echo "❌ Capability quality issues found"
  exit 1
fi

# 警告：发现 TODO
if grep -r "TODO" scripts/; then
  echo "⚠️ Found TODOs (informational only)"
fi
```

---

### 2. Fast Feedback Loop 至关重要

**发现**：
- CI 执行时间直接影响开发体验
- 超过 5 分钟，开发者会切换任务（上下文丢失）
- 超过 10 分钟，开发者会绕过 CI

**org-v2 基准**：
- 总时长：<2 分钟 ✅
- 结构检查：10 秒
- 质量检查：20 秒
- 索引同步：5 秒

**优化策略**：
1. 并行执行（独立检查）
2. 缓存依赖（避免重复下载）
3. 提前失败（结构问题先检查）
4. 跳过不必要步骤（文档 PR 跳过测试）

---

### 3. 错误消息要可操作

**Bad ❌**：
```
Error: Check failed
Exit code 1
```

**Good ✅**：
```
❌ capabilities/INDEX.md is out of sync
   Expected: 12 skills, Found: 10 skills
   
   Fix: Run bash ops/scripts/sync-capability-index.sh
```

**要素**：
- 清晰问题描述
- 期望 vs 实际
- 具体修复命令
- 相关文档链接

---

### 4. Scheduled Workflows 的局限

**尝试**：在 GitHub Actions 中设置定时巡检

**问题**：
- 定时任务不可靠（有时不触发）
- 调试困难（无法本地复现时间触发）
- 上下文缺失（无法访问实时状态）
- 配额消耗（即使没变更也运行）

**结论**：
- ✅ 适合：每日构建、夜间测试、依赖更新检查
- ❌ 不适合：系统巡检、实时监控、状态同步

**替代方案**：
- 本地 cron + 结果提交（org-v2 做法）
- Webhook 触发（事件驱动）
- 手动触发（workflow_dispatch）

---

### 5. 跨平台兼容性是陷阱

**常见问题**：

1. **路径分隔符**
   ```bash
   # Bad
   path="dir/file.txt"  # Windows 上失败
   
   # Good
   path="dir/file.txt"  # 在 Bash 中统一用 /
   ```

2. **行尾符**
   ```bash
   # Bad
   count=$(wc -l file | awk '{print $1}')  # macOS 多空格
   
   # Good
   count=$(wc -l < file | xargs)  # 兼容 macOS/Linux
   ```

3. **命令可用性**
   ```bash
   # Bad
   gsed  # macOS GNU sed
   
   # Good
   sed (使用 POSIX 兼容语法)
   ```

**org-v2 策略**：
- CI 在 ubuntu-latest 运行（标准环境）
- 脚本注释平台要求
- 提供平台特定说明

---

### 6. 权限最小化原则

**发现**：
- 默认 GITHUB_TOKEN 权限过大
- 只读操作不需要写权限
- 减少潜在安全风险

**实践**：
```yaml
permissions:
  contents: read      # 只读代码
  pull-requests: read # 只读 PR
```

仅在需要时请求写权限（如自动提交）

---

### 7. CI 配置也是代码

**最佳实践**：
- [ ] 版本控制
- [ ] Code review（PR 包含 CI 变更）
- [ ] 注释解释（复杂逻辑）
- [ ] 模块化（可复用 action）
- [ ] 测试验证（draft PR 测试）

## 📊 org-v2 CI 架构

### 检查流程

```
PR/Push → GitHub Actions
           ↓
    ┌──────┴──────┐
    │   并行执行   │
    └──────┬──────┘
           ↓
    ┌──────────────┐
    │ 1. 结构检查   │ (10s)
    │ - 必需目录    │
    │ - 必需文件    │
    └──────┬───────┘
           ↓
    ┌──────────────┐
    │ 2. 质量检查   │ (20s)
    │ - 能力格式    │
    │ - 100% 标准   │
    └──────┬───────┘
           ↓
    ┌──────────────┐
    │ 3. 索引同步   │ (5s)
    │ - 统计一致性  │
    └──────┬───────┘
           ↓
    ┌──────────────┐
    │ 4. TODO 扫描  │ (5s)
    │ - 信息提示    │
    └──────┬───────┘
           ↓
         通过/失败
```

### 设计决策

| 决策 | 理由 |
|------|------|
| ubuntu-latest | 标准、快速、免费 |
| 简单 shell 脚本 | 无依赖、易理解、快速 |
| 提前失败 | 快速反馈 |
| 清晰输出 | emoji + 颜色 |
| 最小权限 | 安全 |

## 💡 具体建议

### 对于文档项目

**必须有**：
- 结构检查（必需文件）
- 链接检查（避免 404）
- 拼写检查（可选）

**不需要**：
- 复杂测试
- 构建流程

---

### 对于代码项目

**必须有**：
- 单元测试
- 代码格式（lint）
- 构建验证
- 测试覆盖率

**推荐有**：
- 集成测试
- 安全扫描
- 依赖检查

---

### 对于混合项目（org-v2）

**必须有**：
- 结构验证
- 质量门禁
- 脚本执行测试

**推荐有**：
- 示例项目验证
- 文档链接检查

---

## 🎓 经验教训总结

### 成功因素

1. **快速反馈**：<2 分钟总时长
2. **清晰错误**：可操作的修复建议
3. **渐进增强**：从简单开始，逐步完善
4. **用户体验**：开发者友好的输出

### 失败教训

1. **scheduled workflows 不可靠**：改用本地 cron
2. **过度复杂化**：简单 shell > 复杂框架
3. **警告被忽略**：关键检查必须阻塞

### 意外发现

1. **emoji 提升可读性**：✅ ❌ ⚠️ 一目了然
2. **表格 > 列表**：结构化输出更清晰
3. **错误模板有效**：用户照做就能修复
